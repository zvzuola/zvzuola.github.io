service worker 初体验

发表于 2018-07-06

介绍

service worker是浏览器在后台独立于网页运行的脚本，可以提供离线缓存和消息推送的功能，不能访问DOM和window，可通过self访问全局上下文，通过caches访问cacheStorage。

条件

  • 浏览器支持
  • https(开发过程中可通过localhost使用service worker，上线部署则需要在服务器设置https)

全局caches

service worker通过全局的caches访问cacheStorage，主要用到以下api:

  • caches.open(name)：打开或创建一个名称为name的缓存，返回一个resolve值为该cache的promise。
  • caches.match(request)：检查给定的request是否是CacheStorage对象跟踪的任何Cache对象中的键，如果存在则返回一个以cache为resolve值的promise。
  • caches.keys()：返回一个Promise，resolve的值是cache名称的数组。使用此方法迭代所有Cache对象的列表。
  • caches.delete(name)：删除名称为name的Cache，删除成功返回resolve为ture的promise，否则返回false。

包含三个属性data，height， width

ImageData.data

描述了一个一维数组，包含以 RGBA 顺序的数据，数据使用 0 至 255（包含）的整数表示

ImageData.height

使用像素描述 ImageData 的实际高度

ImageData.width

使用像素描述 ImageData 的实际宽度

sessionStorage cookie的跨域设置以及读取

发表于 2018-05-24

sessionStorage cookie的跨域设置以及读取

问题：在a.com页面有一个字段x,要求从a.com页面跳转到b.com页面后，在b.com页面能获取到字段x

思路：既然要从b.com能获取到字段x，那首先需要把字段x存起来，那么前端存储我们自然想到了cookie，sessionStorage，localStorage等。所以该问题其实就是跨域读取cookie或storage。

作用域

cookie的有效期和作用域

Cookie只受Domian、Path、Expires、HttpOnly、Secure等属性的影响。

  • Domain 标识指定了哪些主机可以接受Cookie。如果不指定，默认为当前文档的主机（不包含子域名）。如果指定了Domain，则一般包含子域名（注意：domain参数可以设置父域名以及自身，但不能设置其它域名，包括子域名，否则cookie不起作用，例如：在a.com下设置cookie: “name=test;domain=b.a.com”，那么在a.com，b.a.com，c.b.a.com下都无法获取cookie）。

  • Path 标识指定了主机下的哪些路径可以接受Cookie（该URL路径必须存在于请求URL中）。以字符 %x2F (“/“) 作为路径分隔符，子路径也会被匹配。

  • Expires 表示cookie的失效时间，如果没有定义，cookie会在对话结束时过期。

  • HttpOnly 表示只发送给服务端，通过JavaScript的 Document.cookie API无法访问。

  • Secure表示cookie只会被https传输。

==由上可知：cookie的作用域并不受端口的影响，http和https下的Cookie也可以共享==
